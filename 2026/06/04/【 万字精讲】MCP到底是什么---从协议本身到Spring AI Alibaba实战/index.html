一、什么是 MCP

1.1 MCP 之前的世界:为什么需要一套新协议

在聊 MCP 是什么之前,先聊聊MCP 出现之前的世界是什么样的——只有搞清楚了"以前有多痛",才能理解 MCP 的设计动机。

2023 ~ 2024 年,大模型刚火起来的时候,所有 AI 应用想用外部工具,都靠一件事:Function Calling。每个 LLM 厂商都有一套自己的 Function Calling 语法:OpenAI 用自己定义的 JSON schema,Anthropic 用 tools 字段,Google 用 functionDeclarations...,格式都不一样。一个典型的痛点场景是这样的:

假设你做了一个 AI 助手,想让它支持 5 个工具(查天气、查日历、发邮件、查 GitHub、写文件),想同时支持 2 个模型(OpenAI、Claude),那么你至少要写:

  • 5 个工具的描述文件 × 2 个模型 = 10 份格式不同的 schema
  • 2 套工具调用适配代码(OpenAI 版、Claude 版)
  • 业务代码和工具调用深度耦合,换一个 LLM,要重写一遍调用层
  • 想再加一个工具、再加一个模型?继续 N × M 组合爆炸

这就是 Function Calling 时代的"集成之痛":M 个 AI 应用 × N 个外部工具,变成 M × N 份胶水代码。每接入一个新工具,都要写一遍定制化的、脆弱的集成代码;每换一个大模型,都要重新适配工具定义。

MCP 想要解决的,正是这个 N × M 的问题——只要大家都遵守同一个协议,工具提供方写一次,所有 AI 应用都能用;AI 应用方接一次,所有支持 MCP 的工具都能用。这就是 USB-C 思路在 AI 时代的复刻:一套标准,所有设备。

1.2 官方定义(英文原文 + 中文翻译)

内容来自官网 : https://modelcontextprotocol.org/docs/getting-started/intro

英文原文:

MCP (Model Context Protocol) is an open-source standard for connecting AI applications to external systems.

Using MCP, AI applications like Claude or ChatGPT can connect to data sources (e.g. local files, databases), tools (e.g. search engines, calculators) and workflows (e.g. specialized prompts)—enabling them to access key information and perform tasks.

Think of MCP like a USB-C port for AI applications. Just as USB-C provides a standardized way to connect electronic devices, MCP provides a standardized way to connect AI applications to external systems.

中文翻译:

MCP(模型上下文协议)是一个用于将 AI 应用连接到外部系统的开源标准。

借助 MCP,像 Claude 或 ChatGPT 这样的 AI 应用,可以连接到数据源(如本地文件、数据库)、工具(如搜索引擎、计算器)和工作流(如专门的提示词模板),从而访问关键信息并执行任务。

可以把 MCP 想象成 AI 应用的 USB-C 接口。正如 USB-C 为电子设备提供了一种标准化的连接方式,MCP 为 AI 应用提供了一种标准化的"连接外部世界"的方式。

对照看,核心就三点:

  • MCP 是什么:一套开源标准,用于把 AI 应用连接到外部系统
  • 能连什么:数据源(本地文件、数据库)、工具(搜索引擎、计算器)、工作流(专门的 Prompt 模板)
  • 如何理解它:AI 应用的 USB-C 接口——一套标准,所有设备

1.3 MCP 能做什么

官网列出的几个典型场景:

  • Agent 可以访问你的 Google Calendar 和 Notion,变成一个更懂你的 AI 助手
  • Claude Code 可以基于 Figma 设计稿,直接生成完整的 Web 应用
  • 企业内的聊天机器人可以连接到组织的多个数据库,让用户用自然语言分析数据
  • AI 模型可以在 Blender 里创建 3D 设计,然后通过 3D 打印机打印出来

1.4 为什么要用 MCP

站在生态不同位置,收益是不一样的:

  • 开发者:MCP 大幅降低了构建或集成 AI 应用/Agent 的开发时间和复杂度——写一次,所有 AI 应用都能用
  • AI 应用/Agent:获得了一个由数据源、工具、应用组成的开放生态,可以增强自身能力、提升用户体验
  • 终端用户:拿到的是更"能干活"的 AI 应用,能访问你的数据,能在你授权下替你完成操作

MCP 已经是事实上的开放协议,被很多主流 AI 工具支持:

  • 国内厂商:百度地图、高德地图、智谱、腾讯 EdgeOne Pages 等都已经官方支持

二、MCP 架构

本节严格依据官网 https://modelcontextprotocol.org/docs/learn/architecture 的结构来讲解。官网把 MCP 的核心概念分成了三块:Participants(参与者)、Layers(分层)、Data Layer Protocol(数据层协议)。下面按这个顺序展开。

2.1 Participants:Host / Client / Server 三者关系

2.1.1 官方定义(英文 + 中文对照)

英文原文(官网):

  • MCP Host:The AI application that coordinates and manages one or multiple MCP clients
  • MCP Client:A component that maintains a connection to an MCP server and obtains context from an MCP server for the MCP host to use
  • MCP Server:A program that provides context to MCP clients

中文翻译:

  • MCP Host(宿主):协调和管理一个或多个 MCP Client 的 AI 应用
  • MCP Client(客户端):维护和某个 MCP Server 的连接,从 MCP Server 拿到上下文给 Host 使用
  • MCP Server(服务端):为 MCP Client 提供上下文的程序

2.1.2 真实关系:一张图看明白

官网给的示意图非常清楚:

这张图的关键信息有四个:

  • 一个 Host 内部可以有多个 Client(图中是 4 个)
  • 每个 Client 维护一条"专用连接(Dedicated connection)" 到某个 Server
  • Server 可以是本地的(Server A、Server B),也可以是远程的(Server C)
  • Host 不直接和 Server 通信,所有通信都通过 Client

官网还举了一个非常具体的例子,帮你把这个关系落地:

Visual Studio Code acts as an MCP host. When VS Code establishes a connection to the Sentry MCP server, the VS Code runtime instantiates an MCP client object that maintains the connection to the Sentry MCP server. When VS Code subsequently connects to another MCP server, such as the local filesystem server, the VS Code runtime instantiates an additional MCP client object to maintain this connection.

翻译:VS Code 本身是 Host。它连 Sentry 时,内部创建一个 Client 维护这条连接;它再连本地文件系统时,内部再创建另一个 Client 维护那条连接。一个 Client 只对应一个 Server,一条专用连接。

2.1.3 一个最容易混淆的认知:"远程"和"本地"到底指什么

很多人第一次接触 MCP 会被一个问题绕进去:我用百度地图的 MCP,是不是我本地的 MCP Client 就直接连到百度地图的服务了?答案是否定的。真实的链路是"本地 MCP Client → 本地 MCP Server(同应用内)→ 远程百度服务",中间多了一层"本地 MCP Server"。我们用你给的百度地图配置来拆解:

这张图的关键信息有三层:

  • 第一层:你的 Java 应用(本机)。它里面有 LLM、有 MCP Client,还有一个被它启动起来的 Node.js 子进程作为"本地 MCP Server"。这三者都在你本机,在同一个应用部署里
  • 第二层:本机 → 远程的边界。只有"本地 MCP Server → 百度地图 HTTP API"这一段是跨网络边界的(走 HTTP,过公网)。前面 Java 客户端和 Node 子进程之间走的是 stdio,完全在本机内,零网络开销
  • 第三层:百度云(远程)。真正的业务能力(地理编码、路线规划、天气查询)在百度云的集群上,它和你的应用完全不在一个网络里

再强调一次,这是最容易搞错的一点:

本地的 MCP Client 永远不直接连远程的百度服务。它只会连本机由 npx 拉起来的那个"本地 MCP Server",然后由这个本地 MCP Server 负责和远程通信。本地 MCP Server 实际上扮演的是一个"远端能力的本地代理"的角色。

为什么 Spring AI 官方推荐的接入方式都是这种"本地代理"模式,而不是 Client 直连远端?

  • 协议解耦:本地 MCP Server 把远端 HTTP/REST/GraphQL 等各种协议统一翻译成 MCP 协议,Client 端只认 MCP
  • 鉴权隔离:API Key 写在本地 MCP Server 的环境变量里,不会泄露到 AI 模型和用户对话中
  • 可观测性:所有远端调用都先经过本地 MCP Server,方便加日志、加限流、加缓存、加熔断
  • 依赖管理:Node.js / Python 版本、uvx / npx 这些运行时环境都由本地 MCP Server 接管,Java 应用不关心
  • 离线可降级:本地 MCP Server 可以加 fallback,远端挂了可以走本地缓存

官网原文佐证(https://modelcontextprotocol.org/docs/learn/architecture#participants):

Note that MCP server refers to the program that serves context data, regardless of where it runs. MCP servers can execute locally or remotely. For example, when Claude Desktop launches the filesystem server, the server runs locally on the same machine because it uses the STDIO transport. This is commonly referred to as a "local" MCP server. The official Sentry MCP server runs on the Sentry platform, and uses the Streamable HTTP transport. This is commonly referred to as a "remote" MCP server.

翻译:MCP Server 这个词指的是"提供上下文数据的服务程序",无论它跑在哪里。

  • 本地 MCP Server:Server 进程跑在本机,用 STDIO 传输(无网络开销)
  • 远程 MCP Server:Server 进程跑在远端(云上或另一台机器),用 Streamable HTTP / SSE 传输(走网络,支持鉴权)

把这一节的核心结论总结成一句话:"本地 Server"指的是 MCP Server 进程部署在你本机,作为你应用的子进程存在;"远程 Server"指的是 MCP Server 进程部署在云上。百度地图这种"远端业务能力 + 本地 stdio 代理"的组合,是 MCP 当前最主流的接入方式。

2.2 Layers:MCP 的两层架构

内容来自官网 https://modelcontextprotocol.org/docs/learn/architecture#layers

英文原文:

MCP consists of two layers:

  • Data layer: Defines the JSON-RPC based protocol for client-server communication, including lifecycle management, and core primitives, such as tools, resources, prompts and notifications.
  • Transport layer: Defines the communication mechanisms and channels that enable data exchange between clients and servers, including transport-specific connection establishment, message framing, and authorization.

Conceptually the data layer is the inner layer, while the transport layer is the outer layer.

中文翻译:

MCP 由两层组成:

  • 数据层(Data layer):定义基于 JSON-RPC 的客户端-服务端通信协议,包括生命周期管理,以及核心原语(如 tools、resources、prompts、notifications)
  • 传输层(Transport layer):定义客户端和服务端之间数据交换的通信机制和通道,包括传输相关的连接建立、消息分帧和授权

概念上,数据层是内层,传输层是外层。

2.2.1 Data Layer(数据层)

数据层实现的是基于 JSON-RPC 2.0 的消息交换协议,定义消息结构和语义。官网把数据层的能力分成了四块:

英文原文(官网):

  • Lifecycle management: Handles connection initialization, capability negotiation, and connection termination between clients and servers
  • Server features: Enables servers to provide core functionality including tools for AI actions, resources for context data, and prompts for interaction templates from and to the client
  • Client features: Enables servers to ask the client to sample from the host LLM, elicit input from the user, and log messages to the client
  • Utility features: Supports additional capabilities like notifications for real-time updates and progress tracking for long-running operations

中文翻译 + 详解:

  • 生命周期管理(Lifecycle management):处理客户端和服务端之间的连接初始化、能力协商、连接终止。MCP 是有状态协议,双方必须先"对一下暗号"才能通信
  • 服务端能力(Server features):让 Server 能给 Client 提供核心功能,包括:
    • Tools(工具):供 AI 调用的可执行函数,做"动作"
    • Resources(资源):上下文数据,供 AI "读"
    • Prompts(提示词):交互模板,帮 LLM 构造结构化对话
  • 客户端能力(Client features):让 Server 能反过来调用 Client,做三件事:
    • Sampling(采样):让 Client 去问它背后的 LLM,生成一段内容
    • Elicitation(引出):让 Client 弹个框向用户要更多信息或确认
    • Logging(日志):让 Server 把日志发给 Client,用于调试和监控
  • 实用能力(Utility features):支持实时更新的通知和长时操作的进度跟踪等附加能力

2.2.2 Transport Layer(传输层)

传输层管的是通信通道和鉴权——连接怎么建立、消息怎么分帧、怎么安全通信。官网明确 MCP 支持两种传输机制:

英文原文(官网):

  • Stdio transport: Uses standard input/output streams for direct process communication between local processes on the same machine, providing optimal performance with no network overhead.
  • Streamable HTTP transport: Uses HTTP POST for client-to-server messages with optional Server-Sent Events for streaming capabilities. This transport enables remote server communication and supports standard HTTP authentication methods including bearer tokens, API keys, and custom headers. MCP recommends using OAuth to obtain authentication tokens.

中文翻译 + 详解:

  • Stdio 传输(Stdio transport):用标准输入/输出流在同一台机器上的本地进程间通信。没有网络开销,性能最优。前面百度地图配置里 npx 启动的进程,和你的 AI 应用之间走的就是 stdio
  • 可流式 HTTP 传输(Streamable HTTP transport):用 HTTP POST 传客户端→服务端的消息,可选 Server-Sent Events (SSE) 做流式响应。支持远程 Server,支持标准的 HTTP 鉴权(Bearer Token、API Key、自定义 Header)。官方推荐用 OAuth 获取 token

两个值得理解的坑:

  • 传输层把通信细节和协议层解耦了——同样的 JSON-RPC 2.0 消息格式,在 stdio 和 HTTP 上都能跑,业务代码不用改
  • stdio 只能"同机":它要求 Client 和 Server 跑在同一台机器上、用同一个进程空间通信;不能跨机器,不能水平扩展(每个 Client 都要拉起一个 Server 子进程)。所以简单规则是:单机本机工具用 stdio,跨机器 / 多租户 / 对外提供用 HTTP

2.3 Data Layer Protocol:JSON-RPC 2.0

内容来自官网 https://modelcontextprotocol.org/docs/learn/architecture#data-layer-protocol

英文原文:

A core part of MCP is defining the schema and semantics between MCP clients and MCP servers. Developers will likely find the data layer — in particular, the set of primitives — to be the most interesting part of MCP. It is the part of MCP that defines the ways developers can share context from MCP servers to MCP clients.

MCP uses JSON-RPC 2.0 as its underlying RPC protocol. Client and servers send requests to each other and respond accordingly. Notifications can be used when no response is required.

中文翻译:

MCP 的核心部分是定义 MCP Client 和 MCP Server 之间的模式(schema)和语义。开发者会发现数据层——特别是其中的原语(primitives)集合——是 MCP 最有意思的部分。它定义了开发者如何把上下文从 MCP Server 共享到 MCP Client。

MCP 使用 JSON-RPC 2.0 作为底层 RPC 协议。客户端和服务端互相发送请求并响应。通知(Notifications)用于不需要响应的场景。

JSON-RPC 2.0 是一个非常轻量的 RPC 协议,核心就几件事:

  • 请求必须带 id(用来匹配响应)、method(要调什么)、params(参数)
  • 响应带回 id 和 result(成功)或 error(失败)
  • 通知是没有 id 的请求——它不期待响应,纯粹用于"我告诉你一件事"

MCP 选择了 JSON-RPC 2.0,是因为它足够简单、跨语言、跨传输——所有主流语言都有现成的 JSON-RPC 实现,stdio 和 HTTP 上都能跑。

2.4 Lifecycle Management:连接的生命周期

内容来自官网 https://modelcontextprotocol.org/docs/learn/architecture#lifecycle-management

MCP 是有状态协议,要求严格的生命周期管理。生命周期管理的作用是协商双方都支持的能力。

一个 MCP 连接的生命周期是三步:

sequenceDiagram
    participant C as MCP Client
    participant S as MCP Server
    Note over C,S: 第一步:连接建立
    C->>S: initialize 请求 (带协议版本、能力、客户端信息)
    S-->>C: initialize 响应 (带协议版本、能力、服务端信息)
    Note over C,S: 第二步:就绪确认
    C->>S: notifications/initialized (无 id,不期待响应)
    Note over C,S: 第三步:正常使用 (list / call / notification)
    C->>S: tools/list
    S-->>C: tools[]
    C->>S: tools/call (name, args)
    S-->>C: 工具执行结果
    Note over C,S: 连接关闭时:任意一方发 disconnect

第一步:initialize(能力协商)。Client 发请求,带三样东西:

  • 协议版本(如 "2025-06-18"):双方必须协商出一致的版本,否则断连
  • 能力声明:声明自己支持哪些能力(tools、resources、prompts、sampling、elicitation 等)
  • 客户端信息:名字、版本号,用于调试和兼容

第二步:notifications/initialized。Client 收到响应后,发一个通知告诉 Server "我准备好了"。这是一个 JSON-RPC notification(没有 id,不期待响应)。

第三步:正常工作。双方按协商出的能力互相调用(list、call、notification 等)。

协商失败直接断连。这一步非常关键——它保证了一方不支持的能力,另一方绝不会去调。

2.5 Primitives:MCP 最核心的概念

官网原话:MCP primitives are the most important concept within MCP. They define what clients and servers can offer each other.

原语(Primitives) 定义了"客户端和服务端能互相提供什么"。理解了它,你就理解了"一个 MCP Server 到底在暴露什么、Host 到底在消费什么"。

2.5.1 Server 端可以暴露的三个原语

内容来自官网 https://modelcontextprotocol.org/docs/learn/architecture#primitives

英文原文:

MCP defines three core primitives that servers can expose:

  • Tools: Executable functions that AI applications can invoke to perform actions (e.g., file operations, API calls, database queries)
  • Resources: Data sources that provide contextual information to AI applications (e.g., file contents, database records, API responses)
  • Prompts: Reusable templates that help structure interactions with language models (e.g., system prompts, few-shot examples)

中文翻译 + 详解:

  • Tools(工具):可执行函数,AI 应用可以调用来执行动作。例:文件操作、API 调用、数据库查询。这是最常用的原语,百度地图 MCP 暴露的"查天气""地理编码"都是 Tool
  • Resources(资源):数据源,为 AI 应用提供上下文信息(只读)。例:文件内容、数据库记录、API 响应
  • Prompts(提示词):可复用的模板,帮 LLM 构造结构化的交互。例:系统提示词、Few-shot 示例

每种原语都有一组对应的方法,**用于发现(*/list)、获取(*/get),有的还可以执行(tools/call):

原语 发现 获取 执行
Tools tools/list tools/get(可选) tools/call
Resources resources/list resources/read —
Prompts prompts/list prompts/get —

这种 list + call 的设计是动态的——Server 上的工具可以随时增删,Client 不用重新部署就能感知(配合 2.6 节的 Notifications)。

官网给的具体例子非常形象:一个提供数据库上下文的 MCP Server,它可以:

  • 暴露 Tools 用于查询数据库
  • 暴露一个 Resource 包含数据库的 schema
  • 暴露一个 Prompt 包含 few-shot 示例,演示怎么用这些工具

2.5.2 Client 端可以暴露的原语

MCP 允许 Server 反过来调用 Client,这是它和普通 RPC 框架最大的不同之一:

  • Sampling(采样):Server 可以让 Client 去问它背后的 LLM,生成一段内容。好处是 Server 不用自己集成 LLM SDK,保持和模型无关。用法:sampling/createMessage 请求
  • Elicitation(引出):Server 可以让 Client 弹个框向用户要更多信息或确认。用法:elicitation/create 请求
  • Logging(日志):Server 可以把日志发给 Client,用于调试和监控

这一组原语是 MCP 真正的"双向"特性——一般的 RPC 协议都是 Client→Server 单向调用,MCP 反过来让 Server 也能"借"Client 的 LLM 和用户交互。

2.5.3 跨切面工具型原语

  • Tasks(实验性):把 MCP 请求包成一个"耐久执行"包装器,支持延迟取结果、状态跟踪。适合昂贵计算、工作流自动化、批处理、多步操作

2.6 Notifications:实时同步

内容来自官网 https://modelcontextprotocol.org/docs/learn/architecture#notifications

英文原文:

The protocol supports real-time notifications to enable dynamic updates between servers and clients. For example, when a server's available tools change—such as when new functionality becomes available or existing tools are modified—the server can send tool update notifications to inform connected clients about these changes. Notifications are sent as JSON-RPC 2.0 notification messages (without expecting a response) and enable MCP servers to provide real-time updates to connected clients.

中文翻译:

协议支持实时通知,让 Server 和 Client 之间能动态更新。例如,当 Server 上的可用工具发生变化时(新功能上线、现有工具被修改),Server 可以主动发工具更新通知告诉 Client。通知是 JSON-RPC 2.0 notification 消息(不期待响应),让 MCP Server 能给连接的 Client 提供实时更新。

最常见的几个通知:

  • notifications/tools/list_changed:工具列表变了
  • notifications/resources/list_changed:资源列表变了
  • notifications/prompts/list_changed:提示词列表变了
  • notifications/progress:长任务进度更新

Client 收到这些通知后,会重新调 */list 刷新本地缓存。这样 Server 可以热加载新工具、新资源,不用通知 Client 重新连接。

三、MCP 的局限性与最佳实践

讲完了"它是什么"和"它怎么工作",我们必须客观地聊一下它现在还不够好的地方,以及怎么用好它。MCP 不是银弹,理解它的边界才能在工程上做出正确决策。

3.1 MCP 的局限性

1.协议本身的安全机制还在补全中

MCP 在早期版本里,没有强制的、标准化的身份认证机制。很多实现把 API Key 直接明文写在 env 字段里——比如百度地图的 BAIDU_MAP_API_KEY 之前就是明文写在 JSON 里的。一旦这个 mcpServers.json 泄露,Key 也就泄露了。

关于"MCP Server 让 LLM 执行任意操作"这件事,需要澄清一下:MCP Server 的本质是提供工具和资源,它不自己包含 LLM,也不直接让 LLM 执行什么。它暴露的 Tool 内部逻辑就是一段普通代码——比如百度地图 MCP 的 handleWeather 内部就是 fetch 调百度 HTTP API,然后把结果按 MCP 格式返回。真正"让 LLM 动手"的是 Host 里的客户端代码——LLM 决定调哪个 Tool,Host 内的 MCP Client 真的去调,Tool 执行完把结果回给 LLM。

真正的安全风险在哪:如果一个 MCP Server 暴露的 Tool 是恶意的(比如 Tool 内部偷偷做"删除文件""发送敏感数据到外网""挖矿"),LLM 很难分辨——LLM 只看 Tool 的 description,根本不知道 Tool 内部到底干了什么。协议自身不提供"工具风险分级",这件事要靠 Host 实现(比如工具调用前弹窗让用户确认、限制工具能访问的目录/网络范围)。

提示词注入(Prompt Injection) 也是反复被讨论的风险。MCP 的设计是 LLM 看工具描述决定调什么,如果 Tool 的 description 里藏了恶意指令,LLM 可能被诱导。这不是 MCP 独有的问题,所有 Function Calling 都有,但 MCP 的开放生态让这个问题更突出。

2.它只是一个通信协议,不是开发框架

官网自己说:"MCP focuses solely on the protocol for context exchange—it does not dictate how AI applications use LLMs or manage the provided context."

翻译:MCP 只管"消息怎么传",不管"消息怎么用"。它不强制 ACL、不提供细粒度的访问控制、不管 LLM 的推理循环、不管 Agent 的工作流编排。这些都得你自己在 Host 侧实现,或者用 LangChain、Spring AI、Semantic Kernel 这样的上层框架。

3.工具数量爆炸时,LLM 选择会变差

MCP 鼓励一个 Server 暴露多个工具,但当一个 Host 同时接了十几个 Server、总工具数破百的时候,LLM 在"选哪个工具"这件事上的准确率会显著下降。Context window 会被工具描述塞满,选择变慢、成本变高、幻觉变多。

4.远程 MCP 的运维成本

虽然 mcp.so 已经极大降低了"找服务"的成本,但"用好"还是要面对:

  • 网络抖动:Streamable HTTP 是有状态长连接,断连重连机制要自己实现
  • 鉴权管理:OAuth / API Key 的轮转、撤销、权限最小化,目前没有统一规范
  • 服务发现:没有标准的"我这个 Host 怎么自动发现组织内部有哪些 MCP Server"。Nacos 3.0 正在做这件事(原生支持 MCP 注册/发现),但生态还在早期
  • 可观测性:调用链、日志、计费这些,都还没有生产级的标准方案

5.STDIO 模式的部署限制

stdio 是性能最好的传输,但它要求 Client 和 Server 在同一台机器、用同一个进程空间通信。这意味着:

  • 不能跨机器
  • 很难水平扩展(每个 Client 都要单独拉起一个 Server 子进程)
  • 容器化部署时,Server 必须和 Client 在同一个 Pod

所以"什么时候用 stdio、什么时候用 HTTP/SSE"是一个工程上必须做的决策。简单规则:单机本机工具用 stdio,跨机器/多租户/对外提供用 HTTP。

3.2 最佳实践

基于上面的局限性,以下是经过社区验证的最佳实践:

  • 能力最小化:每个 MCP Server 只做一件事。比如百度地图 MCP 只做地理/路径/搜索,不要顺手把支付也塞进去
  • 工具命名规范化:用 domain_action 格式(如 baidu_map_geocode),不要用 doIt 这种万能名
  • 描述详细、参数严格:工具的 description 和 inputSchema 是 LLM 决定调不调、怎么调的唯一依据,写不清楚,LLM 一定用错
  • 敏感操作二次确认:删除文件、扣款、发送消息这类副作用,必须由 Host 弹窗让用户确认
  • 密钥走环境变量或密钥管理服务:不要硬编码在配置 JSON 里,Git 提交前用 git-secrets 之类的工具扫一遍
  • 本地优先、远程补充:能本地解决的别远程——性能差几十倍,且不依赖外网。远程 MCP 用于"组织级共享能力"或"无本地替代的服务"
  • 不要一次性接太多 Server:控制在 20~50 个活跃工具,超过就开始做工具检索(RAG over tools)而不是全量塞 Prompt

什么时候才需要打包 + stdio 拉子进程?只有当 MCP Server 和 AI Client 拆成两个独立部署的微服务 / 独立应用时,才需要打成 jar、用 mcp-servers-config.json 配 command: java -jar 走 stdio 通信——这种"跨进程"场景下 MCP 协议的 stdio 价值才真正体现出来。同进程内直接注入 Bean 就够了,不必多此一举。

4.4 stdio 模式的核心优势

  • 零网络协议适配:Client 永远只和本地子进程对话,远端协议(HTTPS / GraphQL / gRPC / WebSocket)的差异由本地代理吸收
  • API Key 不进 LLM 上下文:Key 写在本地子进程的 env 里,不会出现在发给大模型的 Prompt 里,既安全又省 token
  • 依赖管理解耦:Node.js / Python 版本、npx / uvx 这些运行时环境全部由本地子进程接管,Java 应用不关心
  • 可观测性统一:所有远端调用都先经过本地子进程,统一加日志、加限流、加缓存、加熔断
  • 跨 AI 客户端复用:同一份 mcp-servers-config.json,可以直接给 Claude Desktop、Cursor、VS Code、自研 Java 应用共用——这就是 MCP USB-C 思路的真正威力

五、总结

回到开头的 USB-C 比喻:MCP 真正改变的不是"AI 能干什么",而是"AI 怎么和世界对话"。在它之前,每个 AI 应用和每个外部系统都是 M × N 的胶水代码;在它之后,只要你的工具按 MCP 规范暴露,任何支持 MCP 的 AI 应用都能用。

回顾三个核心要点:

  • MCP 是什么:Anthropic 提出的开放协议,被广泛生态支持,生态规模超过 21000+ 服务(https://mcp.so/),目标是给 AI 应用一个"USB-C 接口",解决 Function Calling 时代工具集成的"胶水代码之痛"
  • MCP 怎么工作:Host 编排多个 Client,每个 Client 维护一条到 Server 的专用连接;传输层(Stdio / Streamable HTTP)和数据层(JSON-RPC 2.0)解耦;Server 暴露 Tools / Resources / Prompts,Client 暴露 Sampling / Elicitation / Logging;数据流是 initialize → */list → */call → notifications
  • 本地 vs 远程的真相:MCP Client 从不直接跨网络。接入百度地图这种"远端业务能力"时,真实的链路是"本地 MCP Client → 本地 MCP Server(同应用内的 Node 子进程)→ 远程百度 HTTP API"。本地 MCP Server 扮演的是"远端能力的本地代理"角色,负责把远端协议翻译成 MCP 协议
  • MCP 的局限与最佳实践:协议安全机制还在补全、工具爆炸会拖垮 LLM 选择、stdio 部署受限、远程运维没统一标准;能力最小化、命名规范、描述详细、敏感操作二次确认、密钥走密钥管理、本地优先远程补充、工具数控制在 20~50

Spring AI Alibaba 官方推荐的接入方式就是本地 stdio 模式——本机拉起 MCP Server 子进程作为本地代理,业务代码完全不关心远端协议、API Key 隔离、可观测性统一。

最后一句话:MCP 不是万能的,但它是你现在能让 LLM 真正"动手干活"的最务实的协议标准。当你在犹豫"要不要为这个工具集成 MCP"时,默认要;当你在犹豫"要不要接 50 个 Server"时,默认不要
