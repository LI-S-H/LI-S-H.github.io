一把 Redis 分布式锁，踩透四个坑：锁争抢、僵尸锁、锁过期、锁丢失

发表于2026-07-15|更新于2026-07-15|后端开发

Java 的 synchronized 和 ReentrantLock 只能锁住单个 JVM，一旦服务多节点部署，JVM 之间互不可见，锁就完全失效了。本文从"为什么需要分布式锁"切入，按"问题 → 解决 → 引出新问题 → 再解决"的思路，系统讲清楚 Redis 分布式锁要解决的四大核心问题：锁争抢、僵尸锁、锁过期、锁高可用，以及对应的 SET NX EX、TTL 兜底、看门狗、UUID+Lua、RedLock 方案，最后讨论锁粒度和生产实践。

一、为什么需要分布式锁：Java 单机锁的失效

1.1 Java 内置的两种锁

在讲 Redis 分布式锁之前，先回顾 Java 自己的锁。Java 在 JVM 层面提供了两种主流的锁实现：

  • synchronized 关键字：JVM 内置的隐式锁，基于 Monitor 对象实现（对象头中的 Mark Word 标记锁状态），锁的获取和释放由 JVM 自动完成，不需要手动释放。JDK 6 之后引入了偏向锁、轻量级锁、重量级锁的升级机制，性能已经大幅提升。
  • **ReentrantLock**：JUC 包下的显式锁（java.util.concurrent.locks.Lock 接口的实现），基于 AQS（AbstractQueuedSynchronizer）框架实现，需要手动调用 lock() 和 unlock()，支持公平锁、可中断、可超时、多条件变量等高级特性。

这两种锁有一个共同的前提：它们都只在单个 JVM 内生效。锁的状态存在 JVM 内存中（对象头或 AQS 的 state 变量），不同 JVM 之间互不可见。

1.2 单机锁在多节点部署下完全失效

对于客流量较大的服务，单机部署通常撑不住，业内通用的做法是多节点部署 + 负载均衡，把流量分摊到多个服务实例上。问题就出在这里。

以库存扣减为例，假设部署了服务 A 和服务 B 两个节点，前面挂一层负载均衡：

    participant UB as 用户B
    participant LB as 负载均衡

    participant SB as 服务B (JVM-B)
    participant DB as 共享库存

    SB->>SB: synchronized 加锁 ✅ 成功

    end

把这张图按时间轴拆开看就很清楚了：

  • T0：用户 A 和用户 B 几乎同时（毫秒级差距）发起扣减请求。
  • T1：负载均衡把两个请求分别路由到服务 A 和服务 B。
  • T2：服务 A 在 JVM-A 内加 synchronized 锁成功；几乎同一时刻，服务 B 也在 JVM-B 内加 synchronized 锁成功。因为两套 JVM 的内存空间完全隔离，服务 B 根本看不到 JVM-A 上已经有一把锁，所以它视角里"锁是空闲的"，加锁自然成功。
  • T3：两个服务同时进入临界区，都去读库存、扣减、写回，最终同一份数据被重复扣减。

关键点在于 T2 这一步：两个进程加锁的时间差极小，但由于锁状态分别存在各自的 JVM 内存里，彼此不可见，于是双双成功——这就是单机锁在多节点部署下失效的根本原因。

这就是单机锁失效的本质：

单机锁的可见性仅限于一个 JVM，而分布式部署下多个 JVM 之间没有共享的锁状态，导致互斥性被打破。

要解决这个问题，必须有一把所有节点都能看到的锁——这就是分布式锁。

1.3 什么是分布式锁

分布式锁的本质就是在分布式部署的环境下，实现一种跨进程、跨节点的互斥机制：锁只能被分布式系统中的某一个节点的某一个线程获取，其他节点、其他线程全部都得等待。

具体来说，一把合格的分布式锁需要满足：

  • 互斥性：任意时刻，只有一个客户端能持有锁。
  • 可重入性：同一个客户端（线程）在持有锁的情况下可以再次获取同一把锁，避免死锁。
  • 锁超时释放：持有锁的客户端如果崩溃或网络异常，锁最终要能被自动释放，避免死锁。
  • 高可用：锁服务本身要尽量可用，不能因为单个节点挂了导致整个加锁机制瘫痪。
  • 高性能：加锁、解锁要快，不能成为业务瓶颈。

1.4 为什么选 Redis

业内实现分布式锁有几种主流方案：数据库唯一索引、ZooKeeper、etcd、Redis。其中 Redis 是性能最高、使用最广的方案，原因如下：

  • 基于内存操作：所有读写都在内存中完成，加锁、解锁延迟极低（通常在毫秒级甚至微秒级），是 ZooKeeper、etcd 等基于磁盘或复杂协议的方案无法比的。
  • 核心命令单线程执行：Redis 的命令处理是单线程的（Redis 6.0 后网络 IO 多线程，但命令执行仍然单线程），自身不存在并发冲突，天然适合实现互斥操作。
  • 支持原子复合命令：SET key value NX EX seconds 一条命令完成"查锁 + 加锁 + 设过期"，避免窗口期。
  • 支持 Lua 脚本：复杂的"判断 + 删除"等操作可以用 Lua 脚本保证原子性。
  • 支持主从、哨兵、Cluster：自身具备高可用能力，可作为锁服务的基础设施。
  • 生态成熟：Redisson 等客户端已经把看门狗、可重入、RedLock 都封装好了，开箱即用。

性能和易用性兼具，让 Redis 成为分布式锁的事实标准。但 Redis 分布式锁并不是简单地 SETNX 一下就完事，它本身要解决一连串的问题。

二、Redis 分布式锁要解决的四大核心问题

把单机锁搬到 Redis 上，乍一看只要 SETNX 加锁、DEL 解锁就行了，但实际落地会暴露一连串问题。归纳下来，核心就是以下四个：

编号 问题 根因 后果
问题 1 锁争抢 查锁和加锁是两条命令，中间有窗口期 多个线程同时拿到锁，互斥失效
问题 2 僵尸锁 删锁失败，锁一直不释放 整个业务链路卡死，无法继续加锁
问题 3 锁过期 业务没执行完锁就超时释放 锁被别人拿走，且可能误删别人的锁
问题 4 锁存储 单 Redis 节点宕机，锁直接丢失 所有加解锁机制全部失效

下面按"问题 → 方案 → 新问题"的顺序逐步展开。

3.1 问题复现

最朴素的 Redis 分布式锁实现是两条命令：

四、问题 2：僵尸锁——删锁失败导致锁永不释放

4.1 问题复现

线程拿到了锁，执行完业务后，调用 DEL lock_key 释放锁。但这个 DEL 可能失败，导致锁一直不释放，整个业务再也无法加锁——这就是"僵尸锁"。

删锁失败一般有两类原因：

第一类：Redis 侧或网络侧问题

  • 网络抖动，DEL 的请求包丢在网络中。
  • Redis 连接超时（比如 Redis 正在 BGSAVE fork 子进程，主线程阻塞）。
  • Redis 节点临时不可达。

第二类：业务代码问题

  • 业务执行过程中抛出异常，代码中断，根本没有走到 DEL 这一步。
  • 业务抛了异常后虽然走到了 finally，但 finally 里的 DEL 又抛了新异常。

无论哪种情况，结果都是锁没被释放，下一次任何线程来加锁都会失败，整个被这把锁保护的业务链路彻底卡死。比如库存扣减服务，一旦锁成了僵尸锁，所有后续的扣减请求全部阻塞，业务直接瘫痪。

4.2 解决方案：重试 + try-catch-finally + TTL 兜底

针对两类原因，分别处理。

第一类（Redis/网络问题）：重试 + 降级

针对 Redis 或网络侧的临时故障，采用重试机制：

  • 设置最大重试次数，一般 2~3 次就够。
  • 每次重试之间间隔一小段时间（比如 100~200ms），不要立刻重试。因为故障可能是 Redis 正在 fork 子进程、网络拥塞等，需要给它一点恢复时间。
  • 重试仍然失败的话，把"释放锁"这个动作丢到消息队列里异步处理，借助 MQ 的重试机制兜底。
flowchart TD
    Start[业务执行完毕] --> TryDel[尝试 DEL 释放锁]
    TryDel --> Check{成功?}
    Check -- 是 --> Done[完成]
    Check -- 否 --> Retry1[重试第1次, 间隔 200ms]
    Retry1 --> Check1{成功?}
    Check1 -- 是 --> Done
    Check1 -- 否 --> Retry2[重试第2次, 间隔 200ms]
    Retry2 --> Check2{成功?}
    Check2 -- 是 --> Done
    Check2 -- 否 --> MQ[投递到消息队列异步处理]
    MQ --> Done

第三类（本地服务崩溃）：TTL 兜底

如果本地服务整个崩了（比如 OOM、断电），那 finally 也救不了——进程都没了。这时只能依赖 Redis 端给锁设置的 TTL：进程崩了没关系，到了 30 秒锁会自动过期。

这也是问题 1 中 SET ... EX 30 同时设过期时间的原因，TTL 是僵尸锁的最后一道兜底。

注意：这里默认已经解决了"误删别的线程的锁"的问题（用 Lua 脚本删锁，下文会讲）。否则 TTL 一过期，新线程拿到锁后，老线程的 finally 又来 DEL，会把新线程的锁删掉，引出问题 3。

4.3 TTL 的副作用：引出问题 3

到这里僵尸锁的问题解决了，但 TTL 又引出了新的问题：

既然设了 TTL，就可能出现"业务还没执行完，锁就过期了"的情况。

这就是问题 3——锁过期。

五、问题 3：锁过期——业务没完锁就没了

5.1 问题复现

锁过期会引发两个连锁问题：

问题 3.1：锁的时间不够业务处理

由于网络波动、GC 停顿、磁盘 IO 慢等原因，线程 A 的业务阻塞了一段时间，锁在业务执行完之前就超时释放了。线程 B 进来发现锁是空的，成功加锁，开始执行业务。这时线程 A 醒过来继续执行，导致两个线程同时在临界区操作共享数据，互斥性失效。

问题 3.2：误删别的线程的锁

接上面，线程 A 醒来后执行完业务，按惯例调用 DEL lock_key 释放锁——但它删掉的其实是线程 B 的锁（因为线程 A 的锁早就过期了，B 又重新加了一把）。

线程 C 又进来，发现锁被删了，再次成功加锁。线程 B 还在执行，于是线程 B 和线程 C 同时在临界区。系统彻底错乱：

sequenceDiagram
    participant TA as 线程A
    participant R as Redis
    participant TB as 线程B
    participant TC as 线程C
    TA->>R: 加锁成功, TTL=30s
    Note over TA: 业务阻塞 40s（GC、网络等）
    R-->>R: 30s 到, 锁自动过期
    TB->>R: 加锁成功（锁已过期）
    Note over TB: 线程B 开始执行业务
    Note over TA: 线程A 醒来, 继续执行
    TA->>R: DEL lock_key（误删了B的锁!）
    TC->>R: 加锁成功（锁又被删了）
    Note over TB,TC: ❌ B 和 C 同时在临界区, 数据错乱

所以问题 3 要解决两件事：

  1. 锁的时间不够：让锁能跟着业务执行时间动态续期。
  2. 误删别的线程的锁：让锁只属于当前线程，删除前先校验。

5.2 解决方案一：看门狗机制（解决锁续期）

5.2.1 看门狗原理

大名鼎鼎的"看门狗（Watchdog）"机制就是为锁续期而生的。核心思路：

加锁成功后，在后台开启一个独立线程（看门狗），定时检查业务是否还在执行。如果业务还没结束，就给锁续期（重新 EXPIRE），避免锁提前过期。

flowchart LR
    subgraph 业务线程
        T1[加锁成功] --> T2[执行业务逻辑]
        T2 --> T3[业务完成, 释放锁]
    end
    subgraph 看门狗线程
        W1[启动定时任务] --> W2{业务是否完成?}
        W2 -- 否 --> W3[给锁续期 EXPIRE]
        W3 --> W2
        W2 -- 是 --> W4[停止看门狗]
    end
    T1 -.触发.-> W1
    T3 -.通知停止.-> W4

以 Redisson 的实现为例，看门狗的关键参数：

  • lockWatchdogTimeout：默认 30 秒，作为锁的初始过期时间。
  • 看门狗每 lockWatchdogTimeout / 3 = 10 秒 执行一次续期，把锁的过期时间重置为 30 秒。
  • 业务执行完成后，调用 unlock() 会同时停止看门狗。

5.2.2 看门狗的两个边界情况

情况一：业务一直无法完成

如果业务因为 bug、死循环、依赖故障等原因一直无法结束，看门狗就会无限续期，锁永远不释放。这显然不合理。

解决方案是设置一个兜底的最大续期时间（比如 5 分钟），超过这个时间还认为业务没完成，就直接抛异常、停止续期、告警通知运维介入修复，避免无限续期。

情况二：本地服务崩溃

这是看门狗无法解决的场景。看门狗线程和业务线程跑在同一个 JVM 里，一旦本地服务宕机（OOM、断电、进程被 kill），看门狗线程也跟着死了，没法再续期。锁到了 TTL 自动释放。

但这时危险在于：

线程 A 已经操作了一部分数据（比如扣了库存但还没写日志），还没来得及回滚，服务就崩了。锁释放后，线程 B 进来继续操作这"半截数据"，最终数据彻底错乱，无法回滚，造成重大事故。

5.2.3 看门狗之外的补充方案

既然看门狗解决不了崩溃场景，就要从其他角度做兜底：

  1. **业务设计要"短平快"**：尽量不要让分布式锁保护的业务执行时间过长，让业务执行时间远小于锁的 TTL。把耗时的操作（比如发短信、调外部接口）挪到锁外面异步执行。
  2. 做好业务幂等性：依靠幂等机制防止重复处理。即使锁失效导致同一请求被执行两次，结果也是一致的。
  3. 处理中标记：重要业务在拿到锁后，先在共享存储（数据库或 Redis）里把状态标记为"处理中"。即使本地崩了，其他线程进来发现状态是"处理中"也不会贸然处理，需要人工介入或走补偿流程。
  4. 数据库最终兜底：对于库存这种关键数据，数据库层面也要有乐观锁/版本号兜底，不能完全依赖分布式锁。

    TA->>R: GET lock_key

    Note over TA,R: ⚠ 窗口期：A 的锁正好过期, B 加锁成功
    R-->>R: A 的锁过期
    TB->>R: 加锁成功, value=value_B
    TA->>R: DEL lock_key (删了B的锁!)

六、问题 4：锁存储——单节点 Redis 宕机锁丢失

6.1 问题复现

前面所有方案都基于一个隐含假设：Redis 一直可用。但实际上 Redis 也是会挂的。

如果只有一个 Redis 节点存锁，一旦这个节点宕机：

  • 已经加的锁全部丢失。
  • 后续所有加锁、解锁、查锁的请求全部失败。
  • 整个分布式锁机制瘫痪。

有人会说：给 Redis 配主从+哨兵不就行了？但这里有个致命的同步间隙：

Redis 主从复制是异步的。主节点加锁成功后还没来得及同步给从节点就挂了，哨兵把从节点提升为主节点，但新主节点上没有这把锁——锁丢了。

于是两个客户端可以同时加锁成功，互斥性失效。这就是问题 4 的核心。

6.2 解决方案：RedLock 算法

Redis 作者 antirez 针对这个问题提出了 RedLock（Red lock）算法。核心思想是不依赖单个 Redis 节点，而是同时向多个独立的 Redis 节点申请锁。

6.2.1 算法流程

部署至少 5 个（一般推荐 5 个，最少 3 个）相互独立的 Redis 实例（不是主从、不是 Cluster），客户端依次向每个节点申请同一把锁：

flowchart TD
    C[客户端] --> R1[Redis节点1加锁成功]
    C --> R2[Redis节点2加锁成功]
    C --> R3[Redis节点3加锁成功]
    C --> R4[Redis节点4加锁失败]
    C --> R5[Redis节点5加锁失败]

    R1 --> Count{成功节点数 >= 3?}
    R2 --> Count
    R3 --> Count
    R4 --> Count
    R5 --> Count
    Count -- 是 --> OK[加锁成功]
    Count -- 否 --> Fail[加锁失败, 向所有节点释放锁]

完整步骤：

  1. 客户端记录当前时间戳 T1（毫秒）。
  2. 依次向 N 个 Redis 节点发送加锁请求（SET lock_key unique_value NX EX TTL），每个请求都设置一个很短的连接超时时间（比如 50~100ms），避免某个节点卡死拖慢整体。
  3. 统计成功加锁的节点数：
    • 如果成功数 >= N/2 + 1（半数以上），且总耗时 < TTL（锁还有效），则认为加锁成功，锁的实际有效时间为 TTL - (T2 - T1)。
    • 否则加锁失败，向所有节点（包括没加成功的）发送 DEL 释放锁，让其他客户端能尽快拿到锁。
  4. 业务执行完后，向所有节点发送释放锁请求。

6.2.2 部署要求

RedLock 的几个关键约束：

  • 节点相互独立：不能是主从关系，也不能是 Cluster 节点，必须各自独立的 Redis 实例。
  • 跨机房部署：最好分布在不同机房，避免机房级故障。
  • 设置合理的加锁超时：单节点加锁请求的超时一般设为 50~100ms，避免某节点网络抖动卡死整个流程。
  • TTL 要大于锁的有效窗口：要预留足够时间让业务执行。

6.3 RedLock 的问题与争议

RedLock 看似完美，但实际有不少争议和局限：

问题 1：性能损耗

加锁和释放锁都要向 N 个节点发请求，任何几个节点网络抖动都会拖慢整体效率。相比单节点 Redis 锁，性能下降明显。

问题 2：高并发下成功率低

线程冲突激烈时，要同时抢到半数以上节点的锁很难，可能多次重试都失败，导致服务"半天不动，全在抢锁"。

问题 3：GC 停顿/时钟漂移争议

分布式系统专家 Martin Kleppmann 曾撰文质疑 RedLock：客户端发生长 GC STW 时，可能"自以为还持有锁"，实际上锁早已过期被别人拿走，依然会破坏互斥性。RedLock 依赖各节点时钟一致，但分布式系统时钟漂移是客观存在的。这场争论至今没有定论。

实践建议：

  • 对正确性要求极高的场景（比如金融扣款），建议用 ZooKeeper/etcd 这类基于共识算法的方案，或者直接走数据库事务。
  • 对性能要求高、能容忍极端情况下偶尔失效的场景，单 Redis + 看门狗已经足够，没必要上 RedLock。
  • 真要上 RedLock，直接用 Redisson 的 RedissonRedLock，别手搓。

七、被忽略的关键问题：锁粒度

讲了这么多机制，还有一个容易被忽略却极其关键的问题——锁的粒度，也就是锁的范围。

7.1 粒度的影响

锁粒度直接决定并发能力：

  • 粒度太粗：比如锁住整个商品服务的所有业务，所有商品操作串行执行，并发能力极低。
  • 粒度太细：比如按 SKU 维度加锁，并发能力高，但复杂度上升，还可能出现"锁爆炸"。

7.2 锁爆炸

锁爆炸常见于树形/继承树结构的数据。比如商品分类树：

flowchart TD
    Root[根分类] --> A[分类A]
    Root --> B[分类B]
    A --> A1[子分类A1]
    A --> A2[子分类A2]
    B --> B1[子分类B1]
    B --> B2[子分类B2]
    A1 --> A1a[叶子A1a]
    A1 --> A1b[叶子A1b]

如果操作某个子分类时要锁住它所有的子节点（防止并发修改子树），你就要加 N 把锁。锁释放时只要有一把失败，整个子树都被拖住，连锁反应下问题被放大。

这种场景下，与其细粒度锁每个节点，不如直接锁住整棵子树：

  • 加一把锁，逻辑简单，出问题概率低。
  • 即使释放失败，TTL 兜底也只需要等一把锁的过期。
  • 实现成本低，运维成本低。

7.3 选择粒度的原则

  • 根据业务场景选择合适的粒度，不要为了细而细。
  • 高并发热点数据用细粒度锁（比如按 SKU ID 加锁）。
  • 强关联、树形结构用粗粒度锁（整个子树一把锁）。
  • 不滥用分布式锁，能用本地锁解决的不要上分布式锁。

八、生产实践：用 Redisson，别手搓

前面所有的问题和方案，看起来每一步都不复杂，但真要把 SET NX EX、UUID、Lua、看门狗、重试、TTL 兜底全部正确组合到一起，工作量极大，而且很容易踩坑（比如看门狗线程泄漏、Lua 脚本写错、释放锁时连接断开等）。

Redisson 已经把这一切都封装好了：

  • 可重入锁：基于 Hash 结构记录重入次数。
  • 看门狗：lock.lock() 不传 leaseTime 时自动启用，每 10s 续期到 30s。
  • 公平锁、读写锁、信号量：一应俱全。
  • RedLock：RedissonRedLock 直接可用。
  • Lua 脚本：所有加锁、解锁、续期内部都走 Lua，保证原子性。

生产环境强烈建议直接用 Redisson，不要手搓分布式锁。把精力放在业务设计和锁粒度选择上，比重复造轮子有价值得多。

九、总结：Redis 分布式锁演进全景

用一张图把整条演进线串起来：

flowchart TD
    Start[Java 单机锁synchronized / ReentrantLock] --> P0{多节点部署?}
    P0 -- 是 --> Fail1[锁失效, 互斥被打破]
    Fail1 --> UseRedis[引入 Redis 分布式锁]

    UseRedis --> P1[问题1: 锁争抢查锁和加锁之间有窗口期]
    P1 --> S1[方案: SET NX EX 原子加锁]
    S1 --> P2[问题2: 僵尸锁删锁失败锁不释放]
    P2 --> S2[方案: 重试 + try-finally + TTL 兜底]
    S2 --> P3[问题3: 锁过期业务没完锁就没了]
    P3 --> S31[方案1: 看门狗续期]
    P3 --> S32[方案2: UUID+线程ID+Lua 防误删]
    S31 --> P3b[看门狗解决不了崩溃]
    P3b --> S33[方案3: 业务短平快 + 幂等 + 处理中标记]
    S32 --> P4[问题4: 锁存储单节点宕机锁丢失]
    S33 --> P4
    P4 --> S4[方案: RedLock 多节点加锁]
    S4 --> Final[最终: Redisson 一站式封装]

整条链路的核心思路：

  1. 单机锁失效 → 引入 Redis 分布式锁，让所有节点共享锁状态。
  2. 锁争抢 → SET NX EX 原子加锁，消除查锁和加锁之间的窗口期。
  3. 僵尸锁 → 重试 + try-catch-finally + TTL 三重兜底。
  4. 锁过期 → 看门狗动态续期，UUID + Lua 防误删，业务幂等兜底崩溃场景。
  5. 单节点故障 → RedLock 多节点半数加锁，提升锁服务高可用。
  6. 生产落地 → 直接用 Redisson，不重复造轮子。

最后再强调一句：没有银弹。Redis 分布式锁在极端场景下（GC STW、时钟漂移、看门狗崩溃）依然有失效的可能，对于正确性要求极高的业务，必须配合业务幂等、数据库乐观锁、状态机校验等多重防线，而不是把所有宝都压在分布式锁上。
